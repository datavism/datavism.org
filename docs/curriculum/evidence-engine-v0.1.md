# DATAVISM Curriculum Canon v0.1 — The Evidence Engine

**Status:** Draft v0.1  
**Purpose:** Canonical high-level curriculum architecture for DATAVISM.  
**Use:** Network map, curriculum planning, station design, landing page copy, onboarding, Case File structure, future cohort or lab formats.  
**Core idea:** DATAVISM teaches people how to turn hidden systems into public evidence with data, AI and verifiable methods.

---

## 0. Working Thesis

DATAVISM does not teach "data science" in the classical sense.

It does not begin with:

> Python → SQL → Statistics → Machine Learning → Dashboard

It begins with:

> Question → Data → AI → Verification → Evidence → Intervention

The learner is not primarily framed as a data analyst, data scientist or developer.

The learner becomes a **Datavist**:

> A person who uses data, AI and investigative methods to make hidden systems visible.

The curriculum is therefore not a standard educational sequence. It is an **Evidence Engine**.

Each line teaches a different way to turn a hidden system into public evidence.

---

## 1. The Core Curriculum Logic

The DATAVISM map should not be only a visual design artifact. It must represent a coherent learning architecture.

The curriculum operates on three layers:

| Layer | Function |
|---|---|
| **Story Layer** | Lines, stations, GHOST, Pantheon, Data Underground, Case Files |
| **Skill Layer** | The actual competencies: asking, commanding, collecting, structuring, verifying, publishing |
| **Artifact Layer** | What each learner produces at the end of a station or line |

Every station should answer three questions:

1. **What hidden system are we looking at?**
2. **What method does the learner acquire?**
3. **What evidence artifact does the learner produce?**

---

## 2. The Central Curriculum Formula

> **G teaches the method.**  
> **K traces surveillance.**  
> **R follows power.**  
> **B decodes behavior.**  
> **V reads the future.**  
> **Every line ends in a Case File.**

Or, even shorter:

> **Every line turns a different hidden system into public evidence.**

---

## 3. The Map Progression

The lines should not be five unrelated themes. They need a shared grammar.

Each thematic line follows the same learning arc:

| Station Position | Function |
|---|---|
| **1 — Orientation** | What system are we investigating? |
| **2 — Access** | How do we get signals, sources or data? |
| **3 — Structure** | How do we turn raw material into something analyzable? |
| **4 — Analysis / Verification** | How do we find patterns, contradictions, risks or leverage points? |
| **5 — Case File** | How do we turn the result into public evidence? |

LINE G is the exception and the foundation. It does not investigate one external domain. It teaches the operating system used by all other lines.

---

## 4. Why Five Lines and Five Stations?

The canonical structure is:

> **5 Lines × 5 Stations = 25 Stations**

This should remain stable for the map and curriculum.

Reasons:

- It gives the map a strong visual rhythm.
- It is large enough to feel like a real underground network.
- It is small enough to stay understandable.
- It matches the already defined LINE G structure.
- It creates a modular curriculum: everyone starts with G, then chooses one investigative line.
- It allows future expansion without losing the core map.

---

## 5. Learning Journey

DATAVISM should not require every learner or contributor to complete all lines.

The basic learning path is:

> **LINE G + one thematic line = first full DATAVISM investigation**

Examples:

- `G → K` = investigate surveillance and tracking systems.
- `G → R` = investigate money, ownership and power.
- `G → B` = investigate feeds, attention and behavior.
- `G → V` = investigate long-term systems, climate and future risks.

This makes the map modular but not arbitrary.

Everyone learns the same method first. Then they apply it to a domain.

# 6. LINE G — GHOST / Foundation

## Function

LINE G is the operating system.

It teaches the core DATAVISM method:

> How do I work with AI, data and questions in a way that produces a reliable result?

LINE G is not just an introductory course. It is the initiation line, the methodological backbone and the shared grammar of the entire network.

## Core Question

> How do I turn a vague suspicion into a verifiable investigation?

## Learner Outcome

After LINE G, a learner can:

- formulate a testable question,
- command AI as a co-pilot instead of chatting with it,
- collect and structure raw material,
- verify AI-assisted outputs,
- document a first Case File.

## Map Label

**G — GHOST / Foundation**  
**Ask better. Command cleaner. Verify harder.**

## Station Structure

| Station | Name | Learns | Artifact |
|---|---|---|---|
| **G1** | **THE FOLDER** | Turning diffuse discomfort into a testable data question | Case File #1 |
| **G2** | **COMMAND** | Using AI with specs, constraints, iterations and clear intent | Command Log |
| **G3** | **INTAKE** | Collecting and structuring sources, data and raw material | Intake Sheet / Mini-Pipeline |
| **G4** | **THE CONFIDENT LIE** | Detecting hallucinations, false claims, weak sources and misleading conclusions | Verification Protocol |
| **G5** | **MASCHINENRAUM** | Combining question, command, intake and verification into a complete method | Eval Report |

## Station Notes

### G1 — THE FOLDER

The learner experiences the idea that a system may already have a "folder" on them. The pedagogical move is not to teach a tool first, but to transform discomfort into a precise question.

Key skill:

> A good investigation begins with a question that can be tested.

Artifact:

> **Case File #1** — a first documented question, evidence need and investigation angle.

---

### G2 — COMMAND

The learner stops treating AI as a magic answer box and starts treating it as an operational co-pilot.

Key skill:

> Prompts are not vibes. Prompts are working specs.

Artifact:

> **Command Log** — prompt evolution, constraints, failures, corrections and final useful output.

---

### G3 — INTAKE

The learner learns how to collect, organize and structure raw material: URLs, PDFs, exports, tables, API results, notes, datasets or screenshots.

Key skill:

> Evidence starts as messy intake.

Artifact:

> **Intake Sheet / Mini-Pipeline** — a structured record of sources, fields, metadata and first transformations.

---

### G4 — THE CONFIDENT LIE

The learner practices verifying AI-generated claims and analysis.

Key skill:

> The most dangerous output is not obviously wrong. It is confidently plausible.

Artifact:

> **Verification Protocol** — claim, source, check, status, uncertainty and correction.

---

### G5 — MASCHINENRAUM

The learner combines the full method and sees the "machine room" behind the investigation workflow.

Key skill:

> A DATAVISM investigation is a system: question, command, intake, verification and output.

Artifact:

> **Eval Report** — first complete methodological self-check.

---

# 7. LINE K — KEY / Tracking & OSINT

## Function

LINE K investigates surveillance, tracking, data brokers, identity graphs and OSINT.

It is the line most directly connected to the existing DATAVISM hook:

> They track you. Learn to track back.

## Core Question

> Who watches us, and how?

## Learner Outcome

After LINE K, a learner can:

- identify tracking mechanisms,
- inspect digital signals and traces,
- understand data broker and identity graph logic,
- use public and semi-public sources responsibly,
- document a surveillance or tracking structure as a Case File.

## Map Label

**K — KEY / Tracking & OSINT**  
**Trace the watchers.**

## Pantheon / System

**PANOPTICON**

## Station Structure

| Station | Name | Learns | Artifact |
|---|---|---|---|
| **K1** | **FOOTPRINTS** | Understanding the traces people, browsers, devices and platforms generate | Footprint Map |
| **K2** | **SIGNALS** | Reading pixels, cookies, SDKs, consent flows and request patterns | Tracking Signal Sheet |
| **K3** | **IDENTITY GRAPH** | Understanding IDs, segments, brokers, profiles and entity resolution | Identity Graph Sketch |
| **K4** | **WATCHTOWER** | Monitoring changes, sources, pages, systems or actors with OSINT methods | Watchlist / Monitor |
| **K5** | **PANOPTICON FILE** | Turning a concrete tracking or surveillance structure into public evidence | Panopticon Case File |

## Possible Topics

- Adtech
- Data brokers
- Website tracking
- App SDKs
- Consent theater
- DSA transparency data
- OSINT basics
- Entity resolution with AI
- Browser and platform traces

## Station Notes

### K1 — FOOTPRINTS

The learner maps what kinds of traces are generated by a person, device, browser, app or platform interaction.

Key skill:

> Before you can investigate a tracking system, you need to know what a trace is.

Artifact:

> **Footprint Map** — a visual or tabular inventory of possible traces.

---

### K2 — SIGNALS

The learner inspects technical signals such as pixels, cookies, SDKs, requests, consent flows or event payloads.

Key skill:

> Tracking is not abstract. It leaves signals.

Artifact:

> **Tracking Signal Sheet** — observed signal, source, purpose, owner and evidence.

---

### K3 — IDENTITY GRAPH

The learner understands how fragmented signals can become profiles, segments, IDs or inferred identities.

Key skill:

> The profile is not a fact. It is an assembled claim.

Artifact:

> **Identity Graph Sketch** — actors, IDs, attributes, segments and uncertainty.

---

### K4 — WATCHTOWER

The learner builds a simple monitoring practice for actors, pages, repositories, datasets, policies or infrastructure changes.

Key skill:

> Investigation is not only search. It is watching change over time.

Artifact:

> **Watchlist / Monitor** — defined targets, signals, cadence and change log.

---

### K5 — PANOPTICON FILE

The learner publishes or packages a concrete tracking investigation.

Key skill:

> A tracking finding becomes powerful when it is documented as evidence.

Artifact:

> **Panopticon Case File** — question, sources, method, findings, uncertainty and public output.

---

# 8. LINE R — ROOK / Economy & Power

## Function

LINE R investigates money, ownership, incentives, contracts, markets, lobbying and power.

It follows the oldest investigative principle:

> Follow the money.

## Core Question

> Who benefits?

## Learner Outcome

After LINE R, a learner can:

- map actors and interests,
- read public economic sources,
- structure money, ownership or contract data,
- identify incentives and pressure points,
- document an economic power structure as a Case File.

## Map Label

**R — ROOK / Economy & Power**  
**Follow the value.**

## Pantheon / System

**MAMMON**

## Station Structure

| Station | Name | Learns | Artifact |
|---|---|---|---|
| **R1** | **LEDGER** | Formulating the economic question: who pays, who profits, who carries risk? | Money Question Brief |
| **R2** | **ACTORS** | Mapping companies, people, institutions, owners, funders and lobby actors | Actor Map |
| **R3** | **FLOWS** | Structuring money, contracts, ad spend, subsidies, capital or ownership flows | Flow Table |
| **R4** | **LEVERAGE** | Identifying dependencies, risks, pressure points and power positions | Leverage Map |
| **R5** | **MAMMON FILE** | Turning an economic power structure into public evidence | Mammon Case File |

## Possible Topics

- Platform economics
- Ownership structures
- Public contracts
- Lobby registers
- Corporate networks
- Advertising spend
- Funding flows
- Data economy
- Infrastructure ownership
- Market concentration

## Station Notes

### R1 — LEDGER

The learner learns to translate a moral or political concern into an economic question.

Key skill:

> If a system persists, someone may benefit from it.

Artifact:

> **Money Question Brief** — the economic question, suspected beneficiaries and required evidence.

---

### R2 — ACTORS

The learner identifies the relevant actors: companies, owners, subsidiaries, agencies, funders, regulators or lobby groups.

Key skill:

> Power becomes visible through actors and relationships.

Artifact:

> **Actor Map** — entities, roles, relationships and source references.

---

### R3 — FLOWS

The learner structures economic flows into tables or network representations.

Key skill:

> Flows reveal incentives.

Artifact:

> **Flow Table** — source, target, amount or signal, type, date and confidence.

---

### R4 — LEVERAGE

The learner identifies pressure points, dependencies, risks and structural weaknesses.

Key skill:

> A good investigation does not only show what exists. It shows where power depends on something.

Artifact:

> **Leverage Map** — dependencies, pressure points, risks and possible public-interest angles.

---

### R5 — MAMMON FILE

The learner turns the economic investigation into a public Case File.

Key skill:

> Money evidence needs clarity, restraint and strong sourcing.

Artifact:

> **Mammon Case File** — question, actors, flows, leverage and evidence narrative.

# 9. LINE B — BITE / Feeds, Streams & Behavior

## Function

LINE B investigates feeds, attention, streams, recommendation dynamics, behavioral loops and platform manipulation.

It is the line for systems that shape what people see, repeat, believe or do.

## Core Question

> Why do we keep watching, clicking, scrolling or returning?

## Learner Outcome

After LINE B, a learner can:

- understand feeds and streams as data systems,
- capture or work with event-like data,
- clean and normalize behavioral or content signals,
- detect patterns, peaks, loops or anomalies,
- document a feed or platform dynamic as a Case File.

## Map Label

**B — BITE / Feeds & Behavior**  
**Decode the feed.**

## Pantheon / System

**THE FEED**

## Station Structure

| Station | Name | Learns | Artifact |
|---|---|---|---|
| **B1** | **SOURCE** | Identifying relevant feeds, platforms, channels, APIs, exports or streams | Source Inventory |
| **B2** | **CAPTURE** | Collecting feed, stream, event or content data responsibly | Capture Pipeline |
| **B3** | **NORMALIZE** | Cleaning timestamps, categories, duplicates, formats and event structures | Clean Event Table |
| **B4** | **DETECT** | Finding peaks, patterns, repetitions, anomalies or engagement signals | Signal Report |
| **B5** | **FEED AUTOPSY** | Explaining a platform, feed or behavior dynamic as public evidence | Feed Autopsy Case File |

## Possible Topics

- TikTok, YouTube, Instagram or X mechanics
- Recommendation systems as experienced externally
- Sports betting and gambling content
- Memetic spread
- Trend detection
- Retention loops
- RSS/newsletter/API streams
- Content velocity
- Social listening without surveillance creep

## Station Notes

### B1 — SOURCE

The learner defines which stream matters and why.

Key skill:

> A feed is not the world. It is a selected signal environment.

Artifact:

> **Source Inventory** — feed, source type, access method, limits, risk and relevance.

---

### B2 — CAPTURE

The learner captures data from APIs, exports, scraping, RSS, manual sampling or public feeds.

Key skill:

> Capture defines what can later be seen.

Artifact:

> **Capture Pipeline** — source, method, fields, frequency, constraints and ethical notes.

---

### B3 — NORMALIZE

The learner turns messy feed or event data into consistent structure.

Key skill:

> Streams become analyzable only after normalization.

Artifact:

> **Clean Event Table** — timestamp, source, actor, content, category, metric and notes.

---

### B4 — DETECT

The learner detects patterns, anomalies, loops or signals.

Key skill:

> Attention systems leave rhythms.

Artifact:

> **Signal Report** — pattern, evidence, chart, alternative explanation and confidence.

---

### B5 — FEED AUTOPSY

The learner explains a feed dynamic as a public Case File.

Key skill:

> A feed autopsy turns invisible behavioral design into visible structure.

Artifact:

> **Feed Autopsy Case File** — question, source, capture method, signals, interpretation and limits.

# 10. LINE V — VESPER / Climate, Future & Long Systems

## Function

LINE V investigates long-term systems: climate, demography, infrastructure, environment, migration, resources, public health, urban systems and future risks.

It is slower and more archival than the other lines.

## Core Question

> What changes over time, and what remains when the feed has moved on?

## Learner Outcome

After LINE V, a learner can:

- find and evaluate long-term datasets,
- compare sources across time,
- understand trends, breaks and uncertainty,
- build scenarios or risk narratives,
- communicate long-term change as evidence.

## Map Label

**V — VESPER / Climate & Future**  
**Read the long signal.**

## Pantheon / System

**CUMULUS REX**

## Station Structure

| Station | Name | Learns | Artifact |
|---|---|---|---|
| **V1** | **ARCHIVE** | Finding historical sources, long-term datasets and institutional records | Archive Map |
| **V2** | **PATTERNS** | Reading trends, seasonality, breaks, baselines and long-term movement | Pattern Sheet |
| **V3** | **SCENARIOS** | Working with uncertainty, futures, assumptions and scenario logic | Scenario Board |
| **V4** | **IMPACT** | Connecting long-term patterns to affected groups, risks and consequences | Impact Brief |
| **V5** | **CUMULUS FILE** | Turning a long-system finding into public evidence | Cumulus Case File |

## Possible Topics

- Climate data
- Demographic change
- Migration
- Public health
- Resources
- Urban infrastructure
- Environmental monitoring
- Long-term social risks
- Policy over time
- Historical archives

## Station Notes

### V1 — ARCHIVE

The learner identifies credible historical and long-term data sources.

Key skill:

> Long-term evidence begins with source lineage.

Artifact:

> **Archive Map** — source, institution, time range, variables, access and caveats.

---

### V2 — PATTERNS

The learner analyzes change over time.

Key skill:

> A trend is not a story until its baseline, breaks and uncertainty are understood.

Artifact:

> **Pattern Sheet** — trend, baseline, anomaly, uncertainty and visualization.

---

### V3 — SCENARIOS

The learner works with possible futures without pretending to predict them.

Key skill:

> Scenarios are structured uncertainty, not prophecy.

Artifact:

> **Scenario Board** — assumptions, drivers, scenarios, indicators and warning signals.

---

### V4 — IMPACT

The learner connects long-term signals to affected groups, institutions, risks and consequences.

Key skill:

> Impact is where data becomes socially legible.

Artifact:

> **Impact Brief** — who is affected, how, when, with what uncertainty and what evidence.

---

### V5 — CUMULUS FILE

The learner turns a long-horizon investigation into a public Case File.

Key skill:

> Slow systems need clear evidence because they are easy to ignore.

Artifact:

> **Cumulus Case File** — sources, patterns, scenarios, impact and public narrative.

# 13. Station Design Pattern

Every station should follow the same internal structure.

This prevents the curriculum from becoming a loose collection of interesting topics.

## 1. Signal

A real phenomenon, case, dataset, artifact or public issue.

## 2. Question

The testable investigation question.

## 3. Method

The method or mental model introduced by the station.

## 4. Tooling

AI, spreadsheet, notebook, database, API, scraper, visual tool, search method or manual workflow.

## 5. Verification

How the learner checks whether the result is true, incomplete, misleading or uncertain.

## 6. Artifact

The concrete thing the learner creates.

# 15. Case File Logic

Every line should end in a Case File.

A Case File is the central DATAVISM evidence artifact.

It is simultaneously:

- a learning output,
- a portfolio artifact,
- an investigation record,
- a publication unit,
- a contribution to the Data Underground.

## Canonical Case File Structure

```md
# Case File — [Title]

## 1. Question

What are we trying to prove, test or make visible?

## 2. Context

Why does this matter?

## 3. Sources

What data, documents, APIs, archives or observations were used?

## 4. Method

How was the investigation conducted?

## 5. AI Use

Where was AI used?
For what task?
With what constraints?

## 6. Verification

Which claims were checked?
Against what source?
What remains uncertain?

## 7. Findings

What did we find?

## 8. Limits

What can we not claim?

## 9. Public Output

How is the finding communicated?
Report, visualization, essay, poster, dataset, tool, artwork?

## 10. Next Questions

What should be investigated next?
```

# 17. What DATAVISM Should Not Become

## Not a Tool Course

The lines must not become:

- "Scraping course"
- "SQL course"
- "Prompt course"
- "Dashboard course"

Tools are means, not the identity of the line.

## Not a Generic Data Science Course

DATAVISM should not replicate the classical ladder:

> Python → Pandas → Statistics → ML → Dashboard

That market is already crowded, and it does not fit the brand.

## Not Pure Activism Without Method

DATAVISM should not become outrage without evidence.

Principle:

> No outrage without output.

## Not Pure Education Without Mission

If DATAVISM becomes only "AI data skills", it becomes replaceable.

The mission is the differentiation.

## Not Storyworld Without Skill

GHOST, Pantheon, lines and stations must always map to real capabilities.

Story is the interface. Skill is the substance.

# 18. Strategic Positioning of the Curriculum

DATAVISM can remain intentionally flexible between lab, movement, bootcamp and future commercial training.

The curriculum should therefore be designed as infrastructure, not only as a product.

## If DATAVISM becomes a movement

The lines become investigative desks.

- G = Initiation / Method
- K = Surveillance Desk
- R = Money Desk
- B = Behavior Desk
- V = Future Desk

## If DATAVISM becomes a bootcamp

The lines become curriculum tracks.

- G = Foundation
- K/R/B/V = specialization tracks
- Case File = capstone

## If DATAVISM becomes a lab

The lines become research programs.

- G = Method Lab
- K = Tracking Lab
- R = Power Lab
- B = Feed Lab
- V = Futures Lab

## If DATAVISM later spawns a commercial product

The same method can be repackaged under a different domain for professional or organizational training.

DATAVISM itself can remain the mission-first, open, public-facing canon.
