## 0. Executive Summary

LINE G is the foundation line of DATAVISM.

It is not simply an introduction.  
It is the operating system.

Every other line depends on it.

LINE G teaches the core DATAVISM method:

> Question → Command → Intake → Verification → Evaluation

It turns the learner from a passive consumer of algorithmic systems into someone who can begin an investigation.

LINE G does not start with code.  
It does not start with Python.  
It does not start with a tool tutorial.

It starts with a feeling:

> Something is happening to me, but I cannot yet prove what.

G1 turns that feeling into a question.  
G2 turns that question into a command.  
G3 turns raw material into structured intake.  
G4 tests machine output against evidence.  
G5 evaluates the whole workflow.

After LINE G, the learner is not yet a specialist in tracking, money, feeds or future systems.

But they can enter any line with the right posture:

- ask precisely,
- command AI deliberately,
- collect sources responsibly,
- verify claims,
- document uncertainty,
- and produce a first evidence artifact.

That is why G is the central interchange of the Evidence Engine.

## 1. Role in the Evidence Engine

DATAVISM is an Evidence Engine.

Every thematic line turns a different hidden system into public evidence:

- K traces surveillance.
- R follows power.
- B decodes behavior.
- V reads the future.

But LINE G teaches the method that makes all of that possible.

LINE G answers:

- What exactly are we investigating?
- What would count as evidence?
- How do we use AI without being used by it?
- How do we collect and structure raw material?
- How do we detect false confidence?
- How do we evaluate the investigation process itself?

Without LINE G, the other lines risk becoming:

- tool tutorials,
- aesthetic activism,
- unverified outrage,
- AI-generated overconfidence,
- or interesting but unstructured research.

LINE G prevents that.

It gives DATAVISM its method discipline.

**G — GHOST / Foundation**

GHOST is not a mascot.

GHOST is:

- mentor,
- red-team voice,
- witness,

- anti-hallucination companion,

GHOST does not ask the learner to believe.

## Line Motto

> **Ask better. Command cleaner. Verify harder.**

## Expanded Motto

> Before you investigate the system, learn how not to fool yourself.

## Core Question

> **How do I turn a vague suspicion into a verifiable investigation?**

**The learner's own uncertainty**

In other lines, the opponent is external:

- PANOPTICON,

- vague questions,

## 3. Ethical Boundary

LINE G carries the most important AI-era rule:

> **We do not outsource judgment to the machine.**

AI can assist.

AI can draft.  
AI can classify.  
AI can summarize.  
AI can suggest.  
AI can code.  
AI can search patterns.  
AI can generate hypotheses.

But AI cannot absolve the learner of responsibility.

The person remains accountable for:

- the question,
- the source trail,
- the claim,
- the verification,
- the uncertainty,
- and the public output.

## Not Allowed

LINE G must not become:

- prompt-hack theater,
- AI worship,
- blind automation,
- “just ask ChatGPT” pedagogy,
- productivity content with darker colors,
- fake expertise,
- unverified AI summaries,
- or hallucination laundering.

## Allowed Direction

LINE G should teach:

- careful question design,
- prompts as specs,
- source-aware AI usage,
- structured intake,
- claim checking,
- uncertainty marking,
- evaluation,
- and reproducible evidence work.

## Principle

> The machine can assist the investigation. It cannot become the investigator.

# 6. G1 — THE FOLDER

## Role in the Line

G1 is the threshold station.

It does not begin with a tool.

It begins with the learner discovering that a system may already hold a “folder” about them.

The function of G1 is to transform vague discomfort into a precise question.

The learner moves from:

> “This feels creepy.”

to:

> “What exactly would I need to prove?”

G1 is the emotional and methodological door into DATAVISM.

It establishes that investigation begins before data analysis.

It begins with question design.

## Core Question

> What exactly can be tested?

## Learner Outcome

After G1, the learner can:

- identify a vague suspicion or discomfort;
- turn it into a testable investigation question;
- distinguish broad outrage from investigable claims;
- define what kind of evidence would answer the question;
- mark assumptions;
- create a first Case File.

## Key Concepts

- Investigable question
- Claim
- Evidence need
- Scope
- Actor
- System
- Assumption
- Observable signal
- Public-interest relevance
- Case File

## Suggested Signal / Opening

The learner encounters a “folder” assembled about them by advertising or profiling systems.

The point is not whether the folder is perfectly accurate.

The point is that such systems create claims about people.

The station asks:

> What does the system think it knows?  
> Who creates that claim?  
> What evidence would show how it was produced?

## Method

Question design.

The learner turns a vague concern into a precise, testable question.

## Tooling

Possible tools:

- AI co-pilot for refining questions,
- spreadsheet or notes document,
- public examples of ad categories or profile segments,
- source links,
- manual research,
- simple evidence matrix.

No coding required.

## Artifact

**Case File #1**

Minimum fields:

| Field | Description |
|---|---|
| Suspicion | What feels wrong, strange or important? |
| System | What system may be involved? |
| Actor | Who might create, operate or benefit from it? |
| Question | What exactly are we investigating? |
| Evidence Need | What would answer the question? |
| Source Lead | Where could evidence be found? |
| Assumptions | What is not yet proven? |
| Public Relevance | Why does this matter beyond one person? |

## Verification

The learner must check whether the question is:

- answerable,
- bounded,
- sourceable,
- not purely speculative,
- not framed as a conclusion,
- and not dependent on private or inaccessible data.

## Self-Check

- [ ] I can state the question in one sentence.
- [ ] I can name the system or actor involved.
- [ ] I can describe what kind of evidence would answer it.
- [ ] I can separate suspicion from claim.
- [ ] I can name what I do not know yet.

## Station Sentence

> A feeling is not evidence. But it can become the beginning of a question.

# 7. G2 — COMMAND

## Role in the Line

G2 teaches the learner how to work with AI as a co-pilot.

Not as an oracle.

Not as a search engine with a personality.

Not as a magic expert.

G2 is where the learner stops chatting and starts commanding.

The key shift is:

> prompt → spec

A prompt is not just a question.

In DATAVISM, a command defines:

- task,
- role,
- constraints,
- source boundaries,
- output structure,
- uncertainty handling,
- verification requirements,
- and failure conditions.

G2 is especially important because every later line uses AI.

If G2 is weak, the whole Evidence Engine becomes vulnerable to fluent nonsense.

## Core Question

> How do I make the machine useful without trusting it blindly?

## Learner Outcome

After G2, the learner can:

- write structured AI commands;
- define constraints and output formats;
- ask for uncertainty and assumptions;
- iterate prompts deliberately;
- compare weak and strong commands;
- document AI use in a Command Log;
- avoid outsourcing judgment.

## Key Concepts

- Prompt as spec
- Constraint
- Role framing
- Input context
- Output schema
- Assumptions
- Uncertainty
- Iteration
- AI transcript
- Failure mode
- Verification hook
- Human responsibility

## Suggested Signal / Opening

Show two AI interactions.

### Weak prompt

```text
Tell me about online tracking.
```

### Strong command

```text
You are helping me prepare a public-interest investigation.
Task: turn the following concern into 5 testable questions.
Constraint: do not claim facts without sources.
Output: table with question, evidence needed, possible source, risk of overclaiming.
Uncertainty: mark what cannot be known yet.
Input: I saw that a website loaded multiple adtech domains before I consented.
```

The station asks:

> Which output is more useful, and why?

## Method

Command design.

The learner writes, tests, revises and documents AI commands.

## Tooling

Possible tools:

- AI chat interface,
- command template,
- transcript document,
- spreadsheet for prompt variants,
- source material from G1,
- optional AI comparison across models.

## Artifact

**Command Log**

Minimum fields:

| Field | Description |
|---|---|
| Goal | What did I need the AI to help with? |
| Input | What context did I provide? |
| Command Version | Prompt/spec version |
| Constraints | What did I forbid or require? |
| Output | What did the AI produce? |
| Failure | What was wrong, vague, invented or incomplete? |
| Revision | How did I improve the command? |
| Useful Result | What survived? |
| Verification Need | What still must be checked? |

## Verification

The learner must not treat AI output as evidence.

AI output may be:

- a hypothesis,
- a draft,
- a categorization,
- a planning aid,
- a structuring aid,
- a code suggestion.

It becomes useful only when checked against sources or human reasoning.

## Self-Check

- [ ] I can explain the difference between a prompt and a spec.
- [ ] I can show at least two command iterations.
- [ ] I can name one AI failure or weak output.
- [ ] I can explain what I changed and why.
- [ ] I can mark which AI output still needs verification.

## Station Sentence

> Prompts are not vibes. Prompts are working specs.

# 8. G3 — INTAKE

## Role in the Line

G3 teaches the learner how to collect and structure raw material.

Most investigations fail before analysis.

They fail because the intake is messy:

- links are lost,
- screenshots are unlabeled,
- claims lack source context,
- timestamps are missing,
- PDFs are not tracked,
- data fields are unclear,
- AI summaries replace originals,
- and assumptions get mixed with evidence.

G3 teaches the learner to build the source table before building the argument.

## Core Question

> What raw material do I need, and how do I structure it?

## Learner Outcome

After G3, the learner can:

- collect relevant sources and data;
- preserve source context;
- create a structured intake table;
- identify missing fields;
- use AI to assist structuring without losing originals;
- create a simple mini-pipeline;
- prepare material for verification.

## Key Concepts

- Intake
- Source
- Primary vs. secondary source
- Metadata
- Timestamp
- Field
- Schema
- Raw vs. transformed material
- Evidence trail
- Mini-pipeline
- Data quality
- Provenance

## Suggested Signal / Opening

Take a messy folder of material:

- article links,
- screenshots,
- copied claims,
- policy excerpts,
- PDF downloads,
- table snippets,
- AI summaries,
- notes.

Ask:

> Which of these are sources?  
> Which are claims?  
> Which are transformations?  
> Which could be used as evidence?

## Method

Source intake and structuring.

The learner builds a structured intake sheet.

## Tooling

Possible tools:

- spreadsheet,
- Notion / Airtable / simple database,
- local folder,
- citation manager,
- browser bookmarks,
- AI co-pilot for field extraction,
- manual source review,
- optional notebook for tabular cleaning.

## Artifact

**Intake Sheet / Mini-Pipeline**

Minimum fields:

| Field | Description |
|---|---|
| ID | Stable source or item ID |
| Source Type | Article, PDF, webpage, database, screenshot, request, export etc. |
| Title / Label | Human-readable name |
| URL / Location | Where it came from |
| Accessed At | Date/time accessed |
| Actor | Who published or produced it? |
| Relevant Claim | What claim or evidence does it contain? |
| Raw Material | Link or path to original |
| Extracted Fields | Structured values if any |
| Transformation | What did we do to it? |
| AI Use | Was AI used? How? |
| Confidence | High / medium / low |
| Notes | Caveats or next steps |

## Verification

The learner must preserve raw material.

Rule:

> Do not replace the source with an AI summary.

AI may help extract or organize, but the original must remain accessible.

## Self-Check

- [ ] I can show the original source for each important claim.
- [ ] I can distinguish raw material from transformed material.
- [ ] I can explain my intake fields.
- [ ] I can mark where AI was used.
- [ ] I can identify missing or weak sources.

## Station Sentence

> Evidence starts as messy intake.

# 9. G4 — THE CONFIDENT LIE

## Role in the Line

G4 is the warning station.

It teaches the learner that the most dangerous AI output is not obviously absurd.

The dangerous output is:

- fluent,
- plausible,
- structured,
- helpful-sounding,
- and wrong.

G4 turns skepticism into method.

The learner practices checking claims, finding hallucinations, detecting missing sources and forcing uncertainty into the open.

## Core Question

> Which claims survive contact with the source?

## Learner Outcome

After G4, the learner can:

- identify claims inside AI output;
- separate claim, source and interpretation;
- detect unsupported statements;
- compare AI output against primary sources;
- mark uncertainty;
- correct or discard weak claims;
- create a Verification Protocol.

## Key Concepts

- Hallucination
- Unsupported claim
- Citation mismatch
- Source laundering
- Plausibility trap
- Claim extraction
- Primary source check
- Cross-checking
- Adversarial reading
- Uncertainty marker
- Verification status

## Suggested Signal / Opening

Give the learner an AI-generated explanation of a dataset, article or public issue.

The text should contain:

- true claims,
- unsupported claims,
- subtle overstatements,
- wrong causal interpretations,
- missing uncertainty,
- and maybe one fabricated source.

The learner must not rewrite it.

They must audit it.

## Method

Claim verification.

The learner extracts claims and checks them against sources.

## Tooling

Possible tools:

- AI output transcript,
- source documents,
- spreadsheet,
- verification table,
- secondary AI as red-team assistant,
- browser search,
- source PDFs,
- citation checker.

## Artifact

**Verification Protocol**

Minimum fields:

| Field | Description |
|---|---|
| Claim ID | Stable ID |
| Claim | The exact claim being checked |
| Source Given | What source was cited, if any? |
| Source Checked | What source was actually checked? |
| Evidence | Quote, data point, screenshot, table row etc. |
| Status | Supported / partly supported / unsupported / false / unclear |
| Correction | Better version of the claim |
| Uncertainty | What remains unknown? |
| Risk | Why would this claim be harmful if wrong? |

## Verification

This station is verification.

The key rule:

> A claim is not checked because it sounds right. It is checked because it survives the source.

## Self-Check

- [ ] I can extract individual claims from a fluent text.
- [ ] I can find at least one unsupported or overconfident claim.
- [ ] I can correct a claim without overstating.
- [ ] I can mark uncertainty explicitly.
- [ ] I can explain why verification matters before publication.

## Station Sentence

> The most dangerous lie is the one that sounds like work already done.

# 10. G5 — MASCHINENRAUM

## Role in the Line

G5 is the integration station.

The learner enters the machine room.

They see the full workflow:

```text
Question → Command → Intake → Verification → Evaluation
```

G5 is not about learning a new isolated tool.

It is about evaluating the method itself.

The learner asks:

- Did the question work?
- Did the AI help or mislead?
- Was the intake sufficient?
- Were claims verified?
- What remains uncertain?
- What should improve next time?
- Is the output ready to become public evidence?

## Core Question

> Does the whole machine produce a reliable result?

## Learner Outcome

After G5, the learner can:

- assemble the outputs from G1–G4;
- evaluate the investigation workflow;
- identify weak links;
- assess AI use;
- document remaining uncertainty;
- create an Eval Report;
- decide whether the work can move into a thematic line or Case File.

## Key Concepts

- Workflow evaluation
- Method audit
- Weak link
- Source adequacy
- AI-use disclosure
- Verification coverage
- Uncertainty budget
- Reproducibility
- Publication readiness
- Next investigation

## Suggested Signal / Opening

GHOST shows the learner the machine room.

Not as spectacle, but as audit.

The learner sees all previous artifacts:

- Case File #1,
- Command Log,
- Intake Sheet,
- Verification Protocol.

The station asks:

> Where would this investigation fail if published today?

## Method

Method evaluation.

The learner audits their own investigation workflow.

## Tooling

Possible tools:

- all previous artifacts,
- evaluation checklist,
- AI co-pilot as red-team assistant,
- peer review,
- rubric,
- source matrix,
- publication readiness checklist.

## Artifact

**Eval Report**

Minimum fields:

| Field | Description |
|---|---|
| Investigation Question | Final version |
| Evidence Need | What evidence was required? |
| Command Review | How AI was used and where it failed |
| Intake Review | What sources/material were collected |
| Verification Review | Which claims were checked |
| Weakest Link | Biggest methodological weakness |
| Uncertainty | What remains unknown |
| Ethical Risk | What could cause harm if published |
| Readiness | Not ready / internal only / publishable with caveats / ready |
| Next Step | Continue, revise, expand, or move to thematic line |

## Verification

The learner must identify at least one weakness.

If the Eval Report says everything is perfect, it fails the spirit of G5.

## Self-Check

- [ ] I can explain the full DATAVISM workflow.
- [ ] I can identify the weakest part of my investigation.
- [ ] I can describe how AI helped and where it failed.
- [ ] I can state what remains uncertain.
- [ ] I can decide whether the work is ready for a Case File.

## Station Sentence

> The machine room is where confidence goes to be inspected.
